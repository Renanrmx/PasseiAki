# AGENTS.md

## Project Overview

Passei Aki is a browser extension that records visited pages/downloads and marks links as already visited or partially matched. The source lives in `src/`; generated extension output goes to `dist/`.

Main areas:
- `src/worker/`: background/service-worker logic, persistence, matching, backup/restore, import/export, downloads, support tracking.
- `src/content/highlighter.js`: content script that scans page links and applies visual marks.
- `src/panel/`: popup/settings/history/mirrors/exceptions UI.
- `src/shared/`: scripts shared by worker, panel, and content contexts.
- `src/_locales/*/messages.json`: all user-facing strings.
- `test/pure-functions.test.js`: Node tests that load extension scripts in a VM.

This project uses global browser-extension scripts, not ES modules. Be careful with load order and globals.

## Commands

- `npm test`: run Node tests.
- `npm run lint`: prepare Firefox dist, run `web-ext lint`, and validate Chrome manifest references.
- `npm run prepare:firefox`: build `dist/` for Firefox.
- `npm run prepare:chrome`: build `dist/` for Chrome.
- `npm run build:firefox`: runs tests, prepares Firefox, then builds the package.
- `npm run build:chrome`: runs tests, prepares Chrome, then builds the package.

Before finishing code changes, at minimum run:

```sh
npm test
npm run lint
```

For JS edits, also run `node --check` on the changed files.

Known current lint warnings are from Firefox manifest compatibility and vendor minified `iro.min.js` (`UNSAFE_VAR_ASSIGNMENT`). Do not treat those as newly introduced unless the output changes materially.

## Persistence And Data Safety

User visit data is high-value. Avoid changes that can lose, duplicate, or silently rewrite history incorrectly.

Persistence is normally IndexedDB. When IndexedDB writes are blocked, the extension falls back to in-memory `Map()` stores in `background.database.js`; those data are lost when the extension/browser process closes.

Important persistence points:
- `background.database.js`: IndexedDB schema, memory fallback, exceptions, colors/settings, stats totals.
- `background.mirrors.js`: mirror groups, canonical migrations, global `www.` normalization, atomic memory fallback application.
- `background.backup.js`: backup/restore validation and merge behavior.

When changing migrations or restore/import logic:
- Build a full plan before mutating data.
- Do not mix anonymized (`hashed !== false`) and non-anonymized (`hashed === false`) records in the same merge.
- Merge duplicate visits by summing `visitCount`, keeping the newest `lastVisited`, OR-ing `download`, and safely unioning `queryParamsHash`.
- Keep memory fallback behavior compatible with IndexedDB behavior.
- Preserve backward compatibility with older backups unless explicitly told otherwise.

## URL, Domain, Match, And Mirrors

Central domain helpers are in `src/shared/domains.js`.

Current domain rules:
- Strip only the initial exact `www.` globally.
- `site.com`, `app.site.com`, and `blog.site.com` are different sites.
- HTTP and HTTPS are not distinguished for mirror domain registration.
- Manual domain inputs should normalize protocol/path/query/hash/port away.

Matching and fingerprinting:
- `background.match.js` computes fingerprints and partial/full matches.
- `background.mirrors.js` canonicalizes mirror aliases. First domain in a group is canonical.
- Search should compare a record against virtual alias addresses so partial alias queries can find canonical records.
- Textual search remains limited to readable/non-anonymized records.

## UI And Locales

All user-facing strings should be localized in every `src/_locales/*/messages.json`.

Use existing panel helpers:
- `src/panel/i18n.js` for `data-i18n` and `data-i18n-attr`.
- `src/panel/domain-tags.js` for domain tag inputs used by mirrors and exceptions.
- `src/panel/loading.js` for simple loading/spinner states.

When adding UI text:
- Add a locale key to all languages.
- Prefer existing layout/components/styles.
- Avoid widening popup windows; long text should wrap inside the existing width.

## Tests

Tests use the built-in `node:test` runner and `node:assert/strict`. Do not add a test dependency unless really necessary.

The existing test helper loads browser-extension scripts through `vm`. Objects/arrays from VM contexts may need `toPlain(...)` before `deepEqual`.

Good candidates for tests:
- pure normalization helpers;
- match/fingerprint behavior;
- import/export planning;
- backup validation/merge;
- mirror group normalization/migration;
- memory fallback logic.

Tests are automatically run by `npm run build:chrome` and `npm run build:firefox`.

## Build Output And Generated Files

Do not edit `dist/` directly. It is generated by `scripts/prepare-dist.mjs`.

Avoid committing unrelated generated artifacts unless the task explicitly asks for package/build output.

## Coding Style Notes

- Keep edits scoped to the requested behavior.
- Prefer existing helpers over new abstractions.
- Keep comments rare and useful.
- This codebase primarily uses plain functions and global script files.
- Use ASCII unless the edited file already contains localized/user-facing non-ASCII text.
